# Ad Blocking & Player Reliability

How `userScript.js` strips ads/popunders from phimmoie.fm and keeps the JW Player video working, and what's still known-broken. Written after a long live-debugging session (chrome-devtools MCP against the real production site) that found and fixed four real bugs in sequence — see [History](#history-four-bugs-found-and-fixed) below.

## How injection works

TizenBrew injects this file via CDP's `Page.addScriptToEvaluateOnNewDocument` on the real TV — this re-runs automatically on *every* navigation for the life of that browsing context. The desktop Tampermonkey harness (`tizenphim-web.user.js`) mirrors this with `@match` + `@run-at document-start`, pinning `userScript.js` to a specific commit via `@require` (bump both `@version` and the commit hash together to force a refetch). Both mechanisms guarantee the script is present from the very first instant of every page load, before the site's own JS runs — that's why `document.documentElement`/`body`/`head` can momentarily be `null` (see `whenDocReady()`), and why the ad-hiding CSS below can block ads before they ever paint.

## 1. Ad blocking

- **`WHITELIST`** (regex) — hosts that are part of the site/player and must never be touched: `phimmoie.fm`, `motchille.cx`, `opstream*.com`, `kkphimplayer*.com`, `fcloud.live`, JW Player's own CDN/domains, YouTube (trailers), Google/Cloudflare/reCAPTCHA infra, jsDelivr.
- **`AD`** (regex) — known ad/popunder network name fragments (doubleclick, googlesyndication, popads, adsterra, taboola, adcenter, etc.), matched anywhere in a URL.
- **`isAdUrl(u)`** — the core classifier: whitelisted host → never an ad; matches the `AD` regex → ad; otherwise, *any cross-origin host not on the whitelist* → treated as an ad. That last fallback is deliberately broad (catches ad networks not in the `AD` list) but brittle: every legitimate cross-origin asset the site adds later needs a manual `WHITELIST` addition, or it gets misclassified and hidden. This is exactly what happened with `fcloud.live` (bug #4 below).
- **Document-start CSS** — `AD_SELECTOR` (the site's own ad-container classes/attrs: `.ads-banner`, `ins.adsbygoogle`, `[id^="ad_"]`, `img[src*="adcenter"]`, etc.) gets a `display:none!important` style tag injected as early as `whenDocReady()` allows, so these never render even for a frame.
- **`hide(el)`** — the general-purpose hider. Never calls `.remove()` — phimmoie is a React/Next app and removing a node it still owns throws `removeChild NotFoundError` and can unmount the whole tree (including the player). Instead sets `display/visibility/pointer-events` via inline `!important` and marks `data-tpweb-hidden` so it's not reprocessed. **Important:** `hide()` only hides visually — it does not stop network requests. An already-created ad `<iframe>` keeps loading and executing; if its payload can fire a popunder on load with no click involved, hiding it does not prevent that (see [Known limitations](#known-limitations--gaps)).
- **`hideAdOverlays(root)`** — the site marks every ad-close button with class `no-ads-under` (both the small banner closer and the full-screen ad modal). This hides the nearest ad container, with an explicit guard: never hide anything that *is or contains* the video player, even if it matches the same broad Tailwind classes (`.fixed`, `[class*="inset-0"]`) an ad overlay uses — falls back to hiding just the closer button in that case.
- **`isAdImg(img)` / `killImg(img)`** — images are checked by *known ad host only* (the `AD` regex), not blanket cross-origin blocking, because legitimate movie posters come from CDNs like `img.ophim.live`.
- **`Document.prototype.createElement` patch** — wraps the `.src` property setter on any newly created `<script>`/`<iframe>` so an ad URL assigned via `el.src = url` never loads. **Known gap (Opus review):** this only intercepts the JS property setter — `el.setAttribute('src', url)`, `setAttributeNS`, and elements inserted via `innerHTML` all bypass it entirely. Those ads are only caught afterward by the `MutationObserver` below, which merely hides them (doesn't stop the load).
- **`MutationObserver` + `sweep(root)`** — scans for ad iframes/embeds/img/overlay content already in the DOM and as new nodes get added (childList + attribute mutations on `src`), calling the functions above.

## 2. Popunder blocking

- **`window.open` patch** — replaced with a property descriptor whose getter always returns a no-op function and whose setter is a no-op (an ad script can't restore the native `open()`). **Known gap (Opus review):** the descriptor is `configurable: true`, so a determined page script could `Object.defineProperty`/`delete` it back; and any `<iframe>` on the page gets a fresh, unpatched `contentWindow.open` — the classic bypass. This stops lazy ad scripts, not a determined one.
- **`HTMLAnchorElement.prototype.click` patch** — the *actual, confirmed* popunder mechanism on this site: an ad's click handler creates a real `<a target="_blank" href="...">` and calls `.click()` on it directly, which fires the browser's native new-context behavior independent of `window.open` entirely. This patch refuses to fire `.click()` on any non-whitelisted cross-origin `target="_blank"`/`_new` anchor.
- **`HTMLFormElement.prototype.submit` patch** — same defense for a hidden `<form target="_blank">` submission, a sibling technique. (Opus note: `requestSubmit()` is not patched — minor sibling gap.)
- **`onNav` capture-phase listener** — intercepts clicks/pointer events on `a[href]` globally (bound on both `window` and `document` capture phase, since an ad script's own capture listener on `document` would otherwise win the race). Same-origin links are navigated via `location.assign()` ourselves (stops the site's own handler, which piggybacks a popunder redirect on the same click); non-whitelisted cross-origin links are cancelled outright. **Side effect (Opus note):** this forces every internal link into a full page reload instead of the site's own SPA client-side routing, and `stopImmediatePropagation` on all pointer event types kills any `<a>`-styled control that relied on JS-only (non-navigation) click behavior.

### Known-unfixable popunder class

Documented directly in the script's own comments: the "server"/episode-list buttons (not `<a>` tags) have the ad redirect **fused directly into the same click-handler function body** via `location.href = ...` — confirmed via four separate failed interception attempts (`window.location` setter, `Location.prototype.href`/`assign`/`replace`, CSP `navigate-to`). All of these are either unforgeable browser-platform properties or simply not the actual call site used. There is no known userscript-level trick that can separate "the button's real, intended behavior" from the redirect, because they're the same function call. A second, related class was also observed live: a popunder that fires with **no click or user gesture at all** (an auto-redirect on a timer or page-load trigger) — same root problem (`Location` is unforgeable), different trigger.

## 3. Keeping the JW Player actually playing

- **`killJwAds()`** — watches for the JW Player's own ad-skip button to become clickable (`<div class="jw-skip jw-reset jw-skippable" role="button">` — the `jw-skippable` modifier class is JW's own "now interactable" signal) and dispatches a **real click** on it, exactly mimicking what a real viewer does. Runs for the whole page lifetime (not one-shot), re-arming once the just-clicked button clears the DOM, so a later mid-roll ad break also gets skipped. **Deliberately does NOT call JW's `skipAd()` JS API** — see bug #2 in [History](#history-four-bugs-found-and-fixed).
- **`recoveryWatchdog()`** — gated to only run on watch pages (`location.pathname` matches `/tap-\d+/`; see bug #3). Polls every second for a `<video>` with `readyState >= 3 && currentTime > 0.5`. If nothing has played by 8s, tries clicking a different "server" button (cheap, no reload). If still nothing by 16s, does a capped page reload (max 2 attempts, tracked in `sessionStorage` keyed by episode path so it survives the reload but doesn't loop forever on a genuinely broken episode).

These three timers (`killJwAds` 200ms, `recoveryWatchdog` 1s, the diagnostic badge 300ms, the virtual cursor's keep-alive 1.5s) run **independently with no coordination between them** — don't read any orchestration into the design, there isn't one.

## History: four bugs found and fixed

1. **Fake empty VAST response → JW tears its own player down.** An early version faked an empty-but-valid VAST XML response to the ad-tag request, hoping JW would cleanly report "no ad" and skip straight to content. Confirmed via diagnostic logging this does the opposite: JW's ad plugin treats an empty/adless VAST as an `adError`, and JW's own error-recovery for that specific case tears the whole player instance down (video mounts, `adError` fires, video element vanishes). **Fix:** stopped faking VAST; the real ad request now goes through JW's own, presumably better-tested code path untouched.
2. **`skipAd()` API call → site falls back to an unreachable iframe.** After removing the VAST fake, ads were dismissed by polling JW's `skipAd()` JS API every second from `adStarted`. A controlled A/B test (this whole script present vs. completely absent) proved that specifically *calling the JW API* — not skipping per se — made the site's own logic abandon the native inline player and fall back to a cross-origin iframe embed (`fcloud.live`) that appeared permanently broken. The control run (script absent) played 26+ seconds straight with zero iframe fallback. **Fix:** never touch the JW API for this — click the site's own real skip button instead (see `killJwAds` above).
3. **Recovery watchdog ran everywhere → silent reload loop.** The watchdog added to recover from stuck playback initially had no page-type gate, so it also ran on the homepage and listing pages — which never have a `<video>` element, so "nothing started playing" was always true, and it silently reloaded the page every ~16 seconds regardless of what the user was doing. On a one-shot desktop test tab this reload strips injected protection entirely (looked like "ads reappear if you stop interacting"); on real TV/Tampermonkey deployments the script reinjects fine on reload, but the page would still reload itself unprompted every 16s while a user was just browsing. **Fix:** gated the whole watchdog to only run when the URL matches `/tap-\d+/` (an actual episode/watch page).
4. **Own script hides the real video iframe as an ad.** The "cross-origin iframe = permanently broken" theory from bug #2 turned out to be wrong. Direct inspection proved `fcloud.live` is a legitimate, working video CDN for the site's "Vietsub #1" server — loading that exact iframe URL directly (outside any iframe) played a genuine ~69-minute episode cleanly, and its own network requests showed only JW Player/hls.js/p2p-media-loader infrastructure plus analytics, no ad-network domains. The actual cause of the black screen: `fcloud.live` wasn't in `WHITELIST`, so `isAdUrl()` classified the iframe as an unwhitelisted cross-origin frame — i.e. "an ad" — and the script's own `hide()` call set it to `display:none`. The script was hiding its own legitimate video. **Fix:** added `fcloud\.live` to `WHITELIST`. Confirmed on the same tab, no redirect elsewhere: un-hiding the already-injected iframe in place immediately showed the real movie playing inline, with the site's own UI (rating, episode list, comments) intact around it.

**Methodology note from this bug:** an ad can play through the exact same `<video>` element as the real content, sequentially — a screenshot of "a video is playing" is not proof it's the movie. The only reliable signal is `video.duration` (an ad is seconds-to-tens-of-seconds; a real episode is 40+ minutes) or actually watching the frames long enough to recognize the content.

## Known limitations / gaps

Confirmed unfixable from a userscript (browser platform limitation):
- **`location.href`-fused-into-click-handler popunders** and **zero-click auto-redirect popunders** — both rely on `Location` object properties (`href`, `assign`, `replace`, the `window.location` setter) that are unforgeable in-browser; no known interception point exists for either.

Real, not-yet-fixed gaps identified in a follow-up review (2026-07-19):
- **Recovery watchdog can retrigger the popunder it's trying to route around** — `trySwitchServer()` calls `.click()` on a real site "server" button, which is exactly the control class with the fused, unfixable `location.href` redirect described above. A programmatic `.click()` bypasses `onNav`'s interception (it's not a trusted event and not an `<a href>`), so the site's redirect can fire.
- **Watchdog can false-positive-reload a healthy but autoplay-blocked player** — its "did it play" check requires `currentTime > 0.5`; if the browser blocks autoplay (common in desktop testing, possible on some TV configurations), a fully working, loaded player never advances `currentTime` and gets reloaded (up to twice) as if it were broken.
- **`hide()` doesn't stop network loading** — an ad iframe that's been visually hidden still fetches and executes its own JS; only the (bypassable) `createElement` src-guard prevents the request from firing in the first place.
- **`createElement` src-guard is bypassable** — only intercepts the `.src` *property* setter; `setAttribute('src', ...)`, `setAttributeNS`, and `innerHTML`-inserted elements skip it entirely and fall through to the (load-time-ineffective) `MutationObserver` hide.
- **`window.open` guard is bypassable** — `configurable: true` lets a page script redefine it back; iframes get their own fresh, unpatched `contentWindow.open`.

Design trade-offs, not really bugs:
- The unwhitelisted-cross-origin-iframe-is-an-ad fallback in `isAdUrl()` is deliberately broad to catch unlisted ad networks, but it means **every new legitimate cross-origin asset the site adds requires a manual `WHITELIST` regex update** — this is exactly the shape of bug #4 above, and will recur if the site adds another video CDN or player partner.
- `onNav` forcing every same-origin link through a full `location.assign()` reload means the site's own client-side (SPA) routing never runs — deliberate (keeps ad-blocking active and consistent across navigations without depending on injection timing), but it means no soft transitions.
